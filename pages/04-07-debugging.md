# 레슨 4.7: 디버깅 기법과 팁

---

## 디버깅은 필수 스킬

어떤 워크플로우도 처음부터 완벽하게 작동하지 않습니다. 디버깅(오류 찾기와 수정)은 자동화 개발의 핵심 역량입니다.

좋은 소식: n8n은 디버깅을 위한 탁월한 도구를 제공합니다.

---

## 기본 디버깅 접근법

### 단계 1: 어느 노드에서 문제가 생겼는지 찾기

워크플로우 실행 후:
1. ❌ 빨간색 노드가 문제 노드
2. 해당 노드 클릭 → 에러 메시지 확인

### 단계 2: 입력 데이터 확인

문제 노드의 Input 탭에서:
- 어떤 데이터가 들어왔는지 확인
- 예상한 데이터와 다른지 체크
- null이나 undefined 필드가 있는지 확인

### 단계 3: 설정 확인

- 표현식이 올바른가?
- 크리덴셜이 만료되지 않았는가?
- API 엔드포인트가 정확한가?

---

## n8n 디버깅 도구

### 1. Pin Data (데이터 고정)

특정 노드의 출력 데이터를 고정하여 다음 실행에서 그대로 사용합니다.

**사용법:**
1. 노드 실행 후 Output 탭에서 "Pin" 클릭
2. 이후 실행 시 실제 API를 호출하지 않고 고정된 데이터 사용

**활용:**
- API 호출 없이 반복 테스트
- 특정 케이스(에러 케이스 포함)를 반복 테스트
- 느린 API가 있을 때 개발 속도 향상

### 2. 특정 노드만 실행

특정 노드를 마우스 우클릭 → "Execute Node" 선택:
- 해당 노드만 단독으로 실행
- 이전 노드의 Output을 입력으로 사용
- 전체 워크플로우 실행 없이 단계 테스트

### 3. Workflow Debugger

n8n 1.x 이상에서 제공하는 단계별 디버거:
- 각 노드에 브레이크포인트 설정
- 단계별로 실행하면서 데이터 확인
- 조건 표현식 실시간 평가 확인

## 표현식 디버깅

표현식이 예상대로 작동하지 않을 때:

### 방법 1: 표현식 편집기 활용

필드의 `fx` 클릭 → 표현식 편집기에서 실시간 미리보기 확인

```javascript
{{ $json.user.email }}
// 미리보기: "hong@example.com" 또는 오류 표시
```

### 방법 2: Set 노드로 중간 확인

의심스러운 표현식의 결과를 Set 노드에 담아 확인:

```
[처리 노드] → [Set: debug_value = {{ $json.복잡한.표현식 }}] → [다음 노드]
                         ↓ 결과 확인
```

### 방법 3: Code 노드에서 console.log

```javascript
const data = $input.item.json;
console.log('입력 데이터:', JSON.stringify(data, null, 2));
console.log('이메일 필드:', data.email);

return { json: data };
```

→ n8n 로그에서 console.log 출력 확인 가능

## 자주 발생하는 문제와 해결법

### 문제 1: "Cannot read property 'x' of undefined"

**원인**: null이나 undefined 객체에서 필드를 읽으려 함

```javascript
// 에러 발생
$json.user.email  // user가 null이면 에러

// 해결책
$json.user?.email          // Optional chaining
$json.user?.email ?? ""    // null 처리 추가
```

### 문제 2: 표현식이 그대로 출력됨

```
출력: "{{ $json.name }}"  (표현식 텍스트 그대로)
```

**원인**: 표현식 모드가 아닌 텍스트 모드로 입력됨

**해결**: 필드 오른쪽 `fx` 아이콘 클릭하여 표현식 모드로 전환

### 문제 3: 아이템이 사라짐 (데이터 손실)

**원인**: IF 노드의 False 경로나 Switch의 Default가 연결되지 않음

**해결**: 모든 분기 경로에 노드 연결 (적어도 No Operation)

### 문제 4: 날짜/시간이 틀림

**원인**: 타임존 설정 오류

**해결**:
1. Settings → General → Timezone: Asia/Seoul 확인
2. $now 대신 DateTime.now().setZone('Asia/Seoul') 사용

### 문제 5: API 응답이 비어있음

**원인**: API 호출 시 인증 실패 또는 잘못된 파라미터

**해결**:
1. HTTP Request 노드의 Options → Include Response Headers 활성화
2. statusCode 확인
3. Postman으로 동일한 요청 테스트

## 성능 디버깅

워크플로우가 느릴 때:

### 실행 시간 확인

Executions → 특정 실행 클릭 → 각 노드의 실행 시간 확인

```
[HTTP Request] 15,234ms ← 이 노드가 느림!
[Set]           3ms
[Gmail]         542ms
```

### 병목 해결 방법

| 원인 | 해결책 |
|------|--------|
| 느린 API | 캐싱 추가, 불필요한 호출 줄이기 |
| 대량 데이터 | Split In Batches로 나누어 처리 |
| 직렬 처리 | 가능하면 병렬 처리로 전환 |
| 불필요한 데이터 | 필요한 필드만 선택하여 전달 |

## 디버깅 체크리스트

워크플로우가 작동하지 않을 때 순서대로 확인:

1. [ ] 어느 노드에서 실패했는가?
2. [ ] 해당 노드의 에러 메시지는?
3. [ ] 입력 데이터는 예상대로인가?
4. [ ] 크리덴셜이 유효한가?
5. [ ] API 서비스가 정상인가?
6. [ ] 표현식이 올바른가?
7. [ ] null/undefined 처리를 했는가?
8. [ ] 타임존 설정이 맞는가?

## 핵심 요약

- Pin Data: 특정 노드 출력을 고정하여 반복 테스트
- 특정 노드만 실행: 문제 노드를 단독으로 테스트
- 표현식 편집기: 실시간 미리보기로 표현식 검증
- console.log: Code 노드에서 중간 데이터 출력
- 자주 발생하는 문제: null 처리 누락, 표현식 모드 오류, 분기 경로 누락
