# 레슨 4.1: 효과적인 워크플로우 설계 원칙

---

## 설계 먼저, 구현 나중

많은 사람들이 n8n을 열고 바로 노드를 추가하기 시작합니다. 하지만 이는 좋은 방법이 아닙니다.

올바른 순서:
1. 종이에 흐름을 그린다
2. 필요한 노드와 연결을 정의한다
3. 그 다음 n8n에서 구현한다

---

## 5가지 설계 원칙

### 원칙 1: 목표를 한 문장으로 정의하라

워크플로우를 만들기 전, 다음 형식으로 목표를 적어보세요:

```
"[트리거]가 발생하면, [처리]를 하여, [결과]를 만든다."

예시:
"새 주문이 접수되면(Webhook), 재고를 확인하고 고객에게 확인 이메일을 발송한다."
"매일 오전 9시(Schedule)에, 전날 매출 데이터를 집계하여, 슬랙 채널에 보고한다."
```

이 문장을 쓰지 못한다면, 아직 설계가 덜 된 것입니다.

---

### 원칙 2: 명확한 이름을 붙여라

기본 이름인 "Gmail", "HTTP Request1", "If" 등을 사용하면 나중에 이해하기 어렵습니다.

나쁜 예:
```
[Webhook] → [Set] → [If] → [HTTP Request] → [Gmail]
```

좋은 예:
```
[주문 접수 Webhook] → [데이터 정리] → [VIP 여부 확인] → [재고 API 조회] → [확인 이메일 발송]
```

노드 이름을 지을 때 원칙:
- 무엇을 하는지 (동사 + 목적어)
- 어떤 데이터를 처리하는지
- 2~4단어로 간결하게

---

### 원칙 3: 왼쪽에서 오른쪽으로 흐르게 만들어라

시각적으로 읽기 쉬운 워크플로우를 만드세요.

권장 레이아웃:
```
[트리거] → [처리1] → [처리2] → [결과]
             ↓
          [예외처리]
```

피해야 할 레이아웃:
- 노드가 겹치는 구조
- 화살표가 역방향으로 가는 구조
- 너무 좁은 공간에 노드를 몰아넣는 구조

---

### 원칙 4: 조기 종료로 불필요한 처리를 막아라

데이터 유효성 검사를 초반에 배치하여 잘못된 데이터가 워크플로우 깊숙이 들어오지 못하게 합니다.

나쁜 예 (후반에 체크):
```
[수신] → [처리1] → [처리2] → [처리3] → [이메일 형식 확인] → [발송]
```

좋은 예 (초반에 체크):
```
[수신] → [이메일 형식 확인] → (유효) → [처리1] → [처리2] → [발송]
                              → (무효) → [에러 알림]
```

---

### 원칙 5: 테스트를 위한 여유를 남겨라

실 운영 전에 반드시:
1. 각 노드를 단독으로 테스트 (노드만 선택하여 실행)
2. 엣지 케이스 테스트 (빈 데이터, null 값, 비정상적인 형식)
3. 에러 시나리오 확인 (API 실패, 타임아웃)

---

## 워크플로우 크기 관리

### 언제 분리해야 하나?

워크플로우가 너무 커지면:
- 이해하기 어려워짐
- 디버깅이 복잡해짐
- 재사용이 불가능해짐

20개 노드 이상이 되면 분리를 고려하세요.

### Sub-Workflow로 분리

반복적으로 사용되는 로직은 서브 워크플로우로 분리합니다.

```
메인 워크플로우:
[트리거] → [데이터 준비] → [Execute Workflow: 이메일 발송 로직]

서브 워크플로우 (이메일 발송 로직):
[Webhook 트리거] → [이메일 포맷] → [Gmail 발송] → [로그 기록]
```

## 문서화 습관

### 워크플로우 설명 추가

편집기 → Workflow Details(이름 옆 아이콘) → Description:
- 워크플로우가 무엇을 하는지
- 언제 실행되는지
- 중요한 설정이나 주의사항

### 복잡한 노드에 메모 추가

노드 → Settings → Notes:
```
이 노드는 VIP 고객을 분류합니다.
VIP 기준: 총 구매 금액 500,000원 이상 OR 구매 횟수 10회 이상
2026년 3월 기준 정의 - 변경 시 이 노드 업데이트 필요
```

### Sticky Notes 활용

캔버스에 노란색 메모지(Sticky Note)를 추가하여 워크플로우 전체에 대한 설명을 남길 수 있습니다.

## 설계 체크리스트

워크플로우를 만들기 전 확인:

- [ ] 목표가 한 문장으로 정의되었는가?
- [ ] 트리거는 적절한가?
- [ ] 예외 상황을 모두 고려했는가?
- [ ] 에러 처리 계획이 있는가?
- [ ] 성공 기준을 측정할 수 있는가?
- [ ] 너무 크면 분리할 수 있는가?

---

## 실전 예제: 고객 주문 처리 워크플로우 설계

설계 원칙을 실제 프로젝트에 적용해보는 사례입니다.

### 목표 정의 (원칙 1 적용)

```
"쇼핑몰에서 새 주문이 접수되면(Webhook),
고객 등급과 재고를 확인하여,
등급별 처리(VIP 즉시 처리 / 일반 1일 내 처리)와
확인 이메일을 자동 발송한다."
```

### 전체 구조 설계

```
[주문 접수 Webhook]
    ↓
[필수 데이터 유효성 확인]   ← 원칙 4: 조기 종료
    ├── 실패 → [에러 알림 발송] → (종료)
    └── 성공 ↓
[고객 등급 조회 (DB)]
    ↓
[재고 확인 API 호출]
    ↓
[VIP 여부 분기]
    ├── VIP  → [우선 처리 큐 등록] → [VIP 확인 이메일]
    └── 일반 → [일반 처리 큐 등록] → [일반 확인 이메일]
    ↓
[주문 로그 기록 (Google Sheets)]
```

### 노드 네이밍 적용 (원칙 2)

나쁜 이름 → 좋은 이름:
```
Webhook         → 주문 접수 Webhook
If              → VIP 고객 여부 확인
HTTP Request    → 재고 현황 API 조회
Set             → 주문 데이터 정리
Gmail           → 주문 확인 이메일 발송
```

### 유효성 검사 코드 (원칙 4)

```javascript
// Code 노드: 조기 유효성 검사
const order = $json;
const errors = [];

if (!order.customerId) errors.push('고객 ID 없음');
if (!order.items || order.items.length === 0) errors.push('주문 상품 없음');
if (!order.email || !order.email.includes('@')) errors.push('이메일 형식 오류');
if (!order.totalAmount || order.totalAmount <= 0) errors.push('주문 금액 오류');

if (errors.length > 0) {
  throw new Error(`주문 데이터 오류: ${errors.join(', ')}`);
}

return [{ json: { ...order, validated: true } }];
```

## 실전 예제: 너무 큰 워크플로우 분리하기

**Before**: 노드 25개짜리 단일 워크플로우 (복잡, 재사용 불가)

```
[Trigger] → [파싱] → [유효성 검사] → [DB 조회] → [등급 계산]
         → [이메일 포맷] → [이메일 발송] → [Slack 알림]
         → [시트 기록] → [CRM 업데이트] → [인보이스 생성]
         → [PDF 변환] → [Drive 저장] → [발송 기록] ...
```

**After**: 3개의 서브 워크플로우로 분리

```
메인 워크플로우 (7개 노드):
[Trigger] → [데이터 검증] → [고객 조회]
         → [Execute: 알림 발송 서브]
         → [Execute: 문서 생성 서브]
         → [Execute: CRM 업데이트 서브]

서브 1 - 알림 발송 (5개 노드):
[이메일 발송] + [Slack 알림]

서브 2 - 문서 생성 (6개 노드):
[인보이스 생성] → [PDF 변환] → [Drive 저장]

서브 3 - CRM 업데이트 (4개 노드):
[CRM 업데이트] → [시트 기록]
```

분리 후 효과:
- 메인 로직과 세부 구현 분리로 가독성 향상
- 서브 워크플로우 개별 테스트 가능
- 알림 발송 로직을 다른 워크플로우에서도 재사용 가능

---

## 핵심 요약

- 설계 → 구현 순서를 지킬 것
- 의미 있는 노드 이름 + 왼쪽에서 오른쪽 흐름
- 조기 유효성 검사로 불필요한 처리 방지
- 20개 이상 노드는 서브 워크플로우로 분리
- 문서화는 미래의 나를 위한 투자
- 목표를 한 문장으로 못 쓰면 설계가 덜 된 것
