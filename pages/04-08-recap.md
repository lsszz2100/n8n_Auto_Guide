## 4장에서 배운 것들

4장에서는 좋은 워크플로우를 만들기 위한 실전 기술들을 배웠습니다.

---

## 핵심 개념 요약

### 워크플로우 설계 5원칙
1. 목표를 한 문장으로 정의
2. 의미 있는 노드 이름 사용
3. 왼쪽 → 오른쪽 흐름 유지
4. 조기 종료로 불필요한 처리 방지
5. 테스트 여유 공간 확보

### 조건 처리 요약

| 케이스 | 사용 노드 |
|--------|-----------|
| 참/거짓 2분기 | IF |
| 3개 이상 분기 | Switch |
| 다중 조건 | IF + AND/OR |

### 에러 처리 전략

| 전략 | 언제 사용 |
|------|-----------|
| Stop | 에러 시 즉시 중단 필요 |
| Continue | 에러를 무시하고 계속 |
| Error Output | 에러를 별도로 처리 |
| Error Trigger | 전체 워크플로우 실패 알림 |

### 디버깅 핵심 도구

| 도구 | 용도 |
|------|------|
| Pin Data | 데이터 고정으로 반복 테스트 |
| 노드 단독 실행 | 특정 노드만 테스트 |
| console.log | Code 노드 중간 확인 |
| 실행 기록 | 실제 실행 데이터 추적 |

## 초보자가 자주 하는 실수

### 실수 1: 에러 처리를 나중으로 미룬다

```
잘못된 순서:
  기능 완성 → (나중에) 에러 처리 추가

올바른 순서:
  기능 설계 → 에러 시나리오 파악 → 에러 처리 포함해서 구현
  → 에러는 예외가 아닌 정상적인 흐름의 일부
```

### 실수 2: 루프에서 Rate Limit을 고려하지 않는다

```
잘못된 패턴:
  [Loop Over 1000개] → [API 호출] (Wait 없음)
  → 429 에러, API 차단 위험

올바른 패턴:
  [Loop Over 1000개] → [API 호출] → [Wait: 1초]
  → 안정적인 처리
```

### 실수 3: 분기를 너무 중첩한다

```
피해야 할 구조 (읽기 어려움):
  [IF] → [IF] → [IF] → [처리]

개선 방법:
  Switch 노드 활용 or 서브 워크플로우로 분리
  복잡한 조건 로직은 Code 노드에서 처리
```

## 4장 핵심 패턴 치트시트

### 조건 분기 패턴
```
2가지 경우:      [IF 노드]
3가지 이상:      [Switch 노드]
복잡한 조건:     [Code 노드에서 분류] → [Switch]
```

### 반복 처리 패턴
```
아이템 배열 순회:   [Loop Over Items]
고정 횟수 반복:     [Code 노드 + 카운터]
조건 충족까지:      [Loop] → [IF: 조건 충족?] → 종료/계속
```

### 에러 처리 패턴
```
단순 워크플로우:    Stop and Error (기본값)
중요 데이터 처리:   Error Output → 실패 로그 기록
대량 처리:         Continue + 실패 항목 별도 수집
전체 모니터링:      Error Trigger 워크플로우 별도 운영
```

### 성능 최적화 패턴
```
느린 API:     [Split In Batches] → 배치 처리
대량 데이터:  [Filter] 먼저 → [처리]
병렬 가능:    [병렬 분기] → [Merge]
```
