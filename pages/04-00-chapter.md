## 4장의 목표

3장에서 노드를 배웠다면, 4장에서는 그 노드들을 **올바르게 조합**하는 방법을 배웁니다.

```
멋있어 보이는 워크플로우 ≠ 잘 만들어진 워크플로우
실제로 안정적으로 동작하는 워크플로우 = 잘 만들어진 워크플로우
```

에러 없이 24/7 안정적으로 돌아가는 워크플로우를 만드는 것이 4장의 목표입니다.

## 4장이 필요한 이유

초보자들이 흔히 겪는 문제들:

```
문제 1: 테스트할 때는 됐는데 실제 데이터에서 터진다
문제 2: 에러가 나도 아무도 모른다
문제 3: 어디서 에러가 났는지 찾기 어렵다
문제 4: 100개 아이템을 처리하면 API 속도 제한에 걸린다
문제 5: 워크플로우가 복잡해질수록 수정하기 어려워진다
```

4장은 이 모든 문제를 예방하고 해결하는 방법을 가르칩니다.

## 레슨별 핵심 미리보기

### 4.1 워크플로우 설계 원칙

좋은 워크플로우의 특징:

```
1. 단일 책임: 하나의 워크플로우는 하나의 역할만
2. 명확한 이름: 노드 이름만 봐도 무슨 역할인지 알 수 있도록
3. 에러 경로: 실패했을 때의 경로도 미리 설계
4. 테스트 가능: 실제 데이터 없이도 테스트할 수 있도록
```

### 4.2 트리거 고급 활용

| 트리거 | 사용 시나리오 | 주의사항 |
|--------|-------------|---------|
| Schedule | 매일 오전 9시 리포트 | 시간대(Timezone) 설정 필수 |
| Webhook | 결제 완료 이벤트 수신 | 응답 속도 < 30초 |
| Gmail | 새 주문 이메일 처리 | Polling 간격 설정 |
| Manual | 개발/테스트 시 | 프로덕션에서는 비활성화 |

### 4.3 조건 분기

IF 노드와 Switch 노드를 언제 쓰는지:

```
IF 노드: 두 가지 경우 (true / false)
  예) 이메일이 스팸이면 → 삭제, 아니면 → 처리

Switch 노드: 세 가지 이상 경우
  예) 주문 상태가 → "완료"/"취소"/"환불 요청" 에 따라 다른 처리
```

### 4.4 반복 처리 (Loop)

100개 이메일을 한꺼번에 처리하면 API 제한에 걸립니다.
Loop Over Items 노드로 하나씩, 혹은 묶음으로 처리하는 방법을 배웁니다.

```
배치 크기 설정: 한 번에 10개씩 처리
딜레이 추가:   10개 처리 후 1초 대기
```

### 4.5 실행 모니터링

워크플로우가 활성화된 후:
- Execution 탭에서 실행 이력 확인
- 성공/실패 통계
- 실행 시간 최적화

### 4.6 에러 핸들링

에러가 발생했을 때 워크플로우가 그냥 멈추면 안 됩니다:

```
방법 1: Continue On Fail - 에러를 무시하고 계속 진행
방법 2: Error Trigger    - 에러 발생 시 별도 처리 경로 실행
방법 3: Try/Catch 패턴   - Code 노드에서 예외 처리
```

에러가 나면 Slack/이메일로 즉시 알림을 받는 구조를 배웁니다.

### 4.7 디버깅 기법

문제가 생겼을 때 빠르게 원인을 찾는 방법:

```
1. Execution 탭에서 실패한 실행 클릭
2. 각 노드의 Input/Output 탭으로 데이터 추적
3. 표현식 에디터에서 실시간 결과 미리보기
4. 개별 노드 테스트 실행으로 범위 좁히기
```
