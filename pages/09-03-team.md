## 사용자 관리 (n8n)

### 사용자 초대

```
Settings → Users → Invite User
→ 이메일 입력
→ 역할 선택: Admin / Member
→ 초대 이메일 발송
```

### 역할 별 권한

**Owner (소유자):**
- 모든 기능 접근
- 다른 사용자 관리
- 결제 및 구독 관리

**Admin (관리자):**
- 워크플로우 생성, 편집, 삭제
- 크리덴셜 생성, 공유
- 사용자 초대 (Enterprise)

**Member (멤버):**
- 공유된 워크플로우 접근
- 자신의 워크플로우 생성 및 관리
- 공유된 크리덴셜 사용

## 크리덴셜 공유

팀에서 API 키를 안전하게 공유하는 방법:

```
크리덴셜 생성/편집
→ "Shared with" 섹션
→ 팀원 이름 검색 후 추가

또는
→ "Share with all users" (전체 공유)
```

**크리덴셜 공유 원칙:**
- 공유된 크리덴셜은 API 키 내용을 볼 수 없음 (사용만 가능)
- 민감한 크리덴셜은 필요한 팀원에게만 공유
- 퇴사자 계정 즉시 비활성화 후 API 키 교체

팀 협업을 위한 Git 활용:

```
n8n-workflows/

│   │   └── support-bot.json

│   │   └── lead-scoring.json
│   └── ops/
│       └── inventory-alert.json
├── credentials/
│   └── README.md  # 크리덴셜 목록만 (실제 값 저장 금지!)
└── scripts/
    ├── backup.sh
    └── deploy.sh
```

### 워크플로우 변경 프로세스

```
1. 새 브랜치 생성
   git checkout -b feature/새-이메일-템플릿

2. n8n에서 워크플로우 수정

3. 워크플로우 내보내기
   n8n export:workflow --id=abc123 --output=./workflows/cs/email.json

4. Git 커밋
   git add workflows/cs/email.json
   git commit -m "이메일 템플릿 반품 안내 추가"

5. Pull Request 생성 및 리뷰

6. main 브랜치 머지 후 운영 환경에 배포
   n8n import:workflow --input=./workflows/cs/email.json
```

## CI/CD 파이프라인 연동

n8n 워크플로우 변경을 GitHub Actions로 자동 배포하는 파이프라인:

```yaml
# .github/workflows/deploy-n8n.yml
name: Deploy n8n Workflows

on:
  push:
    branches: [main]
    paths: ['workflows/**']

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
        with:
          fetch-depth: 2

      - name: 변경된 워크플로우 파일 감지
        id: changed
        run: |
          CHANGED=$(git diff --name-only HEAD~1 HEAD -- 'workflows/**/*.json')
          echo "files=$CHANGED" >> $GITHUB_OUTPUT

      - name: 스테이징 환경에 배포
        if: steps.changed.outputs.files != ''
        run: |
          for file in ${{ steps.changed.outputs.files }}; do
            echo "배포 중: $file"
            curl -X POST \
              -H "X-N8N-API-KEY: ${{ secrets.N8N_STAGING_API_KEY }}" \
              -H "Content-Type: application/json" \
              --data @"$file" \
              "${{ secrets.N8N_STAGING_URL }}/api/v1/workflows"
          done

      - name: Slack 배포 알림
        run: |
          curl -X POST ${{ secrets.SLACK_WEBHOOK_URL }} \
            -H 'Content-Type: application/json' \
            -d '{"text":"n8n 워크플로우 스테이징 배포 완료: ${{ steps.changed.outputs.files }}"}'
```

### n8n API로 워크플로우 관리 스크립트

```bash
#!/bin/bash
# scripts/deploy.sh

N8N_URL=$1        # 예: https://n8n.company.com
API_KEY=$2        # n8n API 키
WORKFLOW_FILE=$3  # 배포할 워크플로우 JSON 파일

# 기존 워크플로우 ID 확인
WORKFLOW_NAME=$(jq -r '.name' "$WORKFLOW_FILE")
EXISTING_ID=$(curl -s \
  -H "X-N8N-API-KEY: $API_KEY" \
  "$N8N_URL/api/v1/workflows" | \
  jq -r ".data[] | select(.name == \"$WORKFLOW_NAME\") | .id")

if [ -n "$EXISTING_ID" ]; then
  # 기존 워크플로우 업데이트
  echo "업데이트: $WORKFLOW_NAME (ID: $EXISTING_ID)"
  curl -X PUT \
    -H "X-N8N-API-KEY: $API_KEY" \
    -H "Content-Type: application/json" \
    --data @"$WORKFLOW_FILE" \
    "$N8N_URL/api/v1/workflows/$EXISTING_ID"
else
  # 새 워크플로우 생성
  echo "생성: $WORKFLOW_NAME"
  curl -X POST \
    -H "X-N8N-API-KEY: $API_KEY" \
    -H "Content-Type: application/json" \
    --data @"$WORKFLOW_FILE" \
    "$N8N_URL/api/v1/workflows"
fi
```

## Enterprise 기능

n8n Enterprise 플랜에서 제공하는 팀 협업 기능:

### SSO (Single Sign-On) 설정

```bash
# .env - SAML/OIDC 설정
N8N_AUTH_EXCLUDE_ENDPOINTS=healthz
N8N_USER_MANAGEMENT_DISABLED=false

# SAML 설정 (예: Okta)
N8N_SAML_ENABLED=true
N8N_SAML_ISSUER=https://your-company.okta.com
N8N_SAML_REDIRECT_URL=https://n8n.company.com/rest/sso/saml/callback

# LDAP 설정 (Active Directory)
N8N_LDAP_ENABLED=true
N8N_LDAP_HOST=ldap.company.com
N8N_LDAP_PORT=389
N8N_LDAP_BASE_DN=dc=company,dc=com
N8N_LDAP_BIND_DN=cn=n8n-service,ou=services,dc=company,dc=com
```

### 감사 로그 (Audit Log)

Enterprise에서는 모든 사용자 활동이 기록됩니다:

```
감사 로그에 기록되는 활동:
- 사용자 로그인/로그아웃
- 워크플로우 생성/수정/삭제/활성화
- 크리덴셜 생성/수정/접근
- 사용자 초대/권한 변경
- 실행 기록 조회

조회 방법:
Settings → Audit Logs → 기간 및 이벤트 유형 필터 설정
```

### 프로젝트(폴더) 기능

n8n 1.x Enterprise에서는 워크플로우를 프로젝트로 그룹화하고, 프로젝트별 권한을 별도 관리할 수 있습니다:

```
프로젝트 예시:
  [CS팀 프로젝트]
    ├── 이메일 분류 워크플로우
    ├── 지원 봇 워크플로우
    └── 주간 리포트 워크플로우
    → CS팀원만 접근 가능

  [마케팅팀 프로젝트]
    ├── SNS 자동화
    └── 뉴스레터 발송
    → 마케팅팀원만 접근 가능
```

## 실전 팀 운영 시나리오

### 시나리오 1: 5인 스타트업 팀 운영

```
팀 구성:
- CTO (Owner): 전체 설정, 보안, 비용 관리
- 백엔드 개발자 (Admin): 워크플로우 개발, 크리덴셜 관리
- CS 담당자 (Member): CS 워크플로우 실행 모니터링
- 마케터 (Member): 마케팅 워크플로우 결과 확인
- 인턴 (Member, 읽기 전용): 학습 목적 열람만

운영 원칙:
1. 모든 프로덕션 워크플로우는 Admin만 활성화/비활성화
2. API 크리덴셜은 CTO 또는 개발자만 생성
3. 외부 서비스 연동 전 CTO 승인 필수
4. 주 1회 팀 미팅에서 자동화 성과 공유
```

### 시나리오 2: 30인 팀 역할 분리

```
Level 1 - Owner (CTO/DevOps Lead):
  - n8n 인프라 관리
  - 사용자 권한 최종 승인
  - 비용 및 보안 정책

Level 2 - Admin (팀 리드):
  - 팀 워크플로우 아키텍처 결정
  - 팀원 워크플로우 코드 리뷰
  - 크리덴셜 생성 및 관리

Level 3 - Member (팀원):
  - 담당 도메인 워크플로우 개발
  - 버그 수정 및 개선
  - 실행 결과 모니터링

PR 프로세스:
  개발자 → 스테이징 테스트 → Admin 코드 리뷰 → main 머지 → 운영 배포
```

### 시나리오 3: 워크플로우 인수인계

담당자가 퇴사할 때 안전하게 인수인계하는 절차:

```
1. 담당 워크플로우 전체 목록 작성 (n8n API로 추출)
   curl -H "X-N8N-API-KEY: ..." /api/v1/workflows?ownedBy={userId}

2. 각 워크플로우 문서화 업데이트
   - 트리거 조건, 처리 로직, 의존 시스템, 주의사항

3. 후임자와 1주일 shadow 기간

4. 크리덴셜 소유자 변경 또는 신규 생성
   (퇴사자 계정의 크리덴셜은 비활성화)

5. 퇴사자 계정 비활성화 후 2주 모니터링
   (숨겨진 의존성 파악)
```

## 팀 워크플로우 품질 관리

### 코드 리뷰 체크리스트

PR에서 n8n 워크플로우를 리뷰할 때 확인 사항:

```markdown
## n8n 워크플로우 리뷰 체크리스트

### 기능
- [ ] 명시된 트리거 조건에서 정확히 동작하는가
- [ ] 예외 케이스(빈 데이터, null, 에러)가 처리되어 있는가
- [ ] 테스트 실행으로 동작 확인했는가

### 보안
- [ ] API 키가 하드코딩되지 않고 크리덴셜로 관리되는가
- [ ] 민감한 데이터가 로그에 노출되지 않는가
- [ ] Webhook에 인증이 설정되어 있는가

### 성능
- [ ] SplitInBatches를 사용하여 대량 처리가 안전한가
- [ ] 불필요한 API 호출이 없는가
- [ ] Rate Limit 대응이 있는가

### 유지보수성
- [ ] 워크플로우 명명 규칙을 따르고 있는가
- [ ] 노드에 설명 메모가 있는가
- [ ] 문서화가 업데이트되었는가
```
