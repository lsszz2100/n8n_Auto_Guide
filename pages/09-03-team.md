# 레슨 9.3: 팀 협업과 권한 관리

---

## 팀 규모별 n8n 운영 전략

| 규모 | 구성 | 권장 방식 |
|------|------|-----------|
| 1-2명 | 개인/소규모 | n8n Cloud 기본, 단일 계정 |
| 3-10명 | 스타트업 | n8n Cloud 팀 플랜 또는 Self-hosted |
| 10명 이상 | 기업 | Self-hosted + 역할 관리 |

## 사용자 관리 (n8n)

### 사용자 초대

```
Settings → Users → Invite User
→ 이메일 입력
→ 역할 선택: Admin / Member
→ 초대 이메일 발송
```

### 역할 별 권한

**Owner (소유자):**
- 모든 기능 접근
- 다른 사용자 관리
- 결제 및 구독 관리

**Admin (관리자):**
- 워크플로우 생성, 편집, 삭제
- 크리덴셜 생성, 공유
- 사용자 초대 (Enterprise)

**Member (멤버):**
- 공유된 워크플로우 접근
- 자신의 워크플로우 생성 및 관리
- 공유된 크리덴셜 사용

---

## 크리덴셜 공유

팀에서 API 키를 안전하게 공유하는 방법:

```
크리덴셜 생성/편집
→ "Shared with" 섹션
→ 팀원 이름 검색 후 추가

또는
→ "Share with all users" (전체 공유)
```

**크리덴셜 공유 원칙:**
- 공유된 크리덴셠은 API 키 내용을 볼 수 없음 (사용만 가능)
- 민감한 크리덴셜은 필요한 팀원에게만 공유
- 퇴사자 계정 즉시 비활성화 후 API 키 교체

## 워크플로우 협업 구조

### 팀 역할 분담 예시

```
[자동화 팀 구조]

자동화 리드 (Admin)
├── 워크플로우 아키텍처 설계
├── 공통 모듈 개발
└── 코드 리뷰

개발자 (Member)
├── 새 워크플로우 개발
└── 버그 수정

운영자 (Member)
├── 워크플로우 모니터링
└── 간단한 수정
```

### 워크플로우 명명 규칙

```
[부서]-[기능]-[버전]

예시:
  CS-이메일분류-v2
  SALES-리드스코어링-v1
  OPS-재고알림-v3
  MKT-SNS자동발행-v1
```

### 태그 시스템

```
n8n에서 태그로 워크플로우 분류:

태그 예시:
  #production  - 운영 중
  #staging     - 테스트 중
  #deprecated  - 사용 중단 예정
  #marketing   - 마케팅 팀
  #critical    - 중요 (장애 시 즉시 대응)
```

## Git 기반 워크플로우 관리

팀 협업을 위한 Git 활용:

### 디렉토리 구조

```
n8n-workflows/
├── README.md
├── .gitignore
├── workflows/
│   ├── cs/
│   │   ├── email-classification.json
│   │   └── support-bot.json
│   ├── sales/
│   │   └── lead-scoring.json
│   └── ops/
│       └── inventory-alert.json
├── credentials/
│   └── README.md  # 크리덴셜 목록만 (실제 값 저장 금지!)
└── scripts/
    ├── backup.sh
    └── deploy.sh
```

### 워크플로우 변경 프로세스

```
1. 새 브랜치 생성
   git checkout -b feature/새-이메일-템플릿

2. n8n에서 워크플로우 수정

3. 워크플로우 내보내기
   n8n export:workflow --id=abc123 --output=./workflows/cs/email.json

4. Git 커밋
   git add workflows/cs/email.json
   git commit -m "이메일 템플릿 반품 안내 추가"

5. Pull Request 생성 및 리뷰

6. main 브랜치 머지 후 운영 환경에 배포
   n8n import:workflow --input=./workflows/cs/email.json
```

## 환경 분리 (Dev/Staging/Prod)

```
[개발 환경]         [스테이징]          [프로덕션]
localhost:5678  →  staging.n8n.co  →  app.n8n.co

개발자가           QA가               실제 운영
새 기능 개발        테스트             자동화 실행
```

### 환경별 설정 파일

```bash
# .env.development
N8N_HOST=localhost
DB_TYPE=sqlite
WEBHOOK_URL=https://dev.yourapp.com

# .env.staging
N8N_HOST=staging-n8n.yourcompany.com
DB_TYPE=postgresdb
WEBHOOK_URL=https://staging.yourapp.com

# .env.production
N8N_HOST=n8n.yourcompany.com
DB_TYPE=postgresdb
WEBHOOK_URL=https://yourapp.com
```

## 팀 온보딩 가이드

새 팀원이 n8n을 배울 때 단계적 접근:

### 1주차: 기초 이해
- n8n UI 탐색 (읽기 전용 권한)
- 기존 워크플로우 실행 흐름 이해
- 테스트 워크플로우 만들어보기

### 2-3주차: 실습
- 간단한 워크플로우 수정
- 멘토와 페어 프로그래밍
- 코드 리뷰 참여

### 4주차 이후: 독립 작업
- 새 워크플로우 독립 개발
- Pull Request로 리뷰 받기

## 문서화 관리

팀 협업을 위한 문서화:

```markdown
# 워크플로우 문서화 템플릿

## 워크플로우: [이름]

**목적**: 무엇을 자동화하는가
**트리거**: 어떤 이벤트로 시작되나
**담당자**: 누가 관리하나
**최종 수정**: YYYY-MM-DD
**태그**: #production #cs

## 동작 방식
1. 이메일 수신 시 자동 실행
2. AI가 카테고리 분류
3. 담당자에게 Slack 알림

## 설정 필요 항목
- Gmail 크리덴셜 (CS팀 공용 계정)
- Slack 웹훅 URL (#cs-alerts 채널)

## 주의사항
- 환불 이메일은 반드시 팀장 검토 필요
- 하루 처리 한도: 500건
```
